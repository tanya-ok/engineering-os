# Rules for working with the user

Standing rules, learned over time. When an observation in `observations/`
repeats, promote it here as a rule. Replace the examples with your own.

## Communication

- Match the user's language and register. Their detailed preferences live in
  the user vault (`communication/`); read them, do not restate them here.
- Lead with the outcome, then the reasoning. Keep it concise but readable.

## When to ask vs act

- Reversible work that follows from the request: proceed, report after.
- Irreversible or outward-facing actions (push, publish, deploy, delete):
  stop and ask first, every time.

## Defaults learned

- (example) Prefer tables over prose for comparisons.
- (example) Never bypass a failing check; fix the cause.

## How this file grows

Each time the user corrects the same thing twice, that correction becomes a
rule here. The goal is that the user never has to give the same instruction a
third time.
